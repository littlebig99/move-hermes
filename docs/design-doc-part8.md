## 9. 企微/飞书机器人集成

### 9.1 企微机器人回调

```
基本流程：
1. 用户在企业微信群 @机器人 或发送图片
2. 企微推送消息到我们的回调URL
3. 后端接收 → 保存图片 → 调用AI识别
4. 识别结果写入数据库 → 更新订单/工序状态
5. 机器人回复确认消息
```

### 9.2 飞书机器人回调

```
基本流程：
1. 订阅事件：im.message.receive_v1
2. 用户发送图片到群
3. 飞书推送事件到回调URL
4. 后端处理 → AI识别 → 更新进度
5. 机器人回复确认消息
```

## 11. 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| AI识别准确率不足 | 用户信任崩塌 | 始终保留人工复核入口；初期对低置信度结果强制人工确认 |
| U盘读写速度慢 | 用户体验差 | SQLite启用WAL模式；减少频繁写入；批量合并写入 |
| 断电/异常退出 | 数据损坏 | SQLite WAL模式自带原子性；每次写入后同步；定期备份 |
| 企微/飞书API变更 | 功能不可用 | 抽象适配器层；多平台兼容；降级为手动上传 |
| 用户不会配置API | 无法使用 | 提供预设模板；内置推荐模型；简化配置流程 |
| 数据安全问题 | 客户流失 | 数据仅存储在U盘本地；不上传任何业务数据；可选加密 |

## 12. 工程评审意见

### 架构评审

**优点：**
1. 技术栈选择合理 — FastAPI + SQLite 最适合便携部署场景
2. 零构建步骤的前端方案 — 对U盘分发至关重要
3. 数据与程序分离 — U盘上 data/ 目录独立于程序目录

**风险项：**
1. SQLite在U盘上的性能 — 需要测试不同U盘速度下的响应时间
2. 企微/飞书回调URL需要公网可达 — 但服务是本地运行的
   - **解决方案：** 使用内网穿透（ngrok/frp）或让用户在局域网内访问
3. 多人同时写入SQLite — 需要测试WAL模式下的并发表现

### 改进建议

1. 增加健康检查端点 `/health` 用于U盘拔出检测
2. 配置向导应支持"推荐模式"一键配置，减少用户操作步骤
3. 生产日志的AI识别结果应缓存，避免重复调用API浪费费用
4. 考虑增加数据自动备份到U盘另一个分区的功能
